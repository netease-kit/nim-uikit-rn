# IM2 Feature Implementation

用于把 im2-rn-demo 里“当前功能是怎么实现的”抽出来，复用成后续功能开发的标准路径。

目标：
- 优先复用当前仓库已有实现模式
- 快速定位该功能应该落在哪些 route / store / service / utils
- 明确最小可落地实现路径
- 给出足够具体的测试用例
- 在需要时先产出 spec，再进入实现

## 何时使用

出现以下任一类请求时，直接使用本 skill：
- “按当前项目的方式实现 XX 功能”
- “先看这个仓库里类似功能怎么做的”
- “参考现在项目现有能力补齐某个模块”
- “把参考项目的功能迁到当前仓库”
- “先把这个功能的 spec 写出来”
- “给出当前仓库里的核心实现路径和测试点”
- “补齐某个页面/流程/交互/接口能力”

如果用户一次提多个功能，先拆成功能清单，再逐个处理；不要把多个功能混成一个大方案。

如果仓库内存在以下材料，优先读取并吸收：
- 测试用例文档，例如 `TestCases.xlsx`
- `AGENTS.md`
- `ARCHITECTURE.md`
- `README.md`
- 参考项目或历史实现
- 已有 spec / proposal / design / tasks
- 当前模块的真实代码路径

如果存在“参考实现”和“测试用例”冲突，优先以测试用例为准。

如果存在“参考实现”和“当前仓库既有边界”冲突，优先保持当前仓库边界，再决定如何迁移。

## 工作顺序

### 1. 先界定功能边界

优先从以下来源收集事实：
- `AGENTS.md`
- `ARCHITECTURE.md`
- 当前功能所属 route 文件
- 当前功能所属 store 文件
- 当前仓库真实代码
- 已有 spec / 设计文档 / 测试用例
- 参考项目实现
- 用户显式说明的验收口径

至少澄清这些问题；如果能从上下文直接确定，就不要反问用户：
1. 这个功能解决什么用户问题
2. 功能入口和出口是什么
3. 影响哪些页面、路由、store、接口、状态
4. 是否有参考实现或测试用例优先级
5. 是否先出 spec，还是直接实现

### 2. 找出核心实现路径

不要只给抽象建议。必须基于真实代码路径，指出：
- 入口层：页面、路由、命令入口、事件入口
- 状态层：store、hook、context、缓存、派生状态
- 数据层：接口、SDK、数据库、消息流、数据转换
- 视图层：组件树、交互节点、空态/错误态/加载态
- 验证层：单测、集成测试、手工验证路径

如果仓库中已有近似能力，优先复用并明确“扩展点”；不要默认重写。
如果你不确定功能该落在哪里，优先查 `references/project-implementation-map.md`。
如果已经明确是登录、会话、聊天、好友、群组或设置类功能，优先直接读取对应模块 reference。

### 2.1 固定定位顺序

在这个仓库中，默认按下面顺序定位：
1. 先看 `AGENTS.md` 和 `ARCHITECTURE.md`，确认边界和 OpenSpec 门槛
2. 再看拥有该行为的 route 文件
3. 再看对应 store
4. 再看 service / utils / constants
5. 最后再看参考项目如何迁移进来

不要一上来直接在组件里塞业务逻辑，也不要绕过 store 直接在页面里堆 SDK 调用。

### 3. 拆成最小实现步骤

按“能交付、能验证”的粒度拆解，而不是按技术层机械拆解。

优先使用这种拆法：
1. 建立功能入口和最小可见 UI
2. 接通核心数据流
3. 补齐关键状态变更和错误处理
4. 完成边界交互和兼容逻辑
5. 补齐测试与验收说明

每一步都要能回答：
- 改哪些文件
- 为什么改这些文件
- 完成后用户能看到什么变化

如需固定格式，优先参考：
- `references/spec-template.md`
- `references/testcase-template.md`
- `references/project-implementation-map.md`
- `references/login-implementation.md`
- `references/conversation-implementation.md`
- `references/chat-implementation.md`
- `references/friend-implementation.md`
- `references/team-implementation.md`
- `references/settings-implementation.md`

## 当前仓库实现原则

在 im2-rn-demo 中，默认遵守这些实现原则：
- 路由入口归 `app/`，不要把导航结构下沉到通用组件
- 页面负责展示和交互编排，业务状态优先放在 store
- NIM SDK 初始化和生命周期归 `stores/NIMStore.ts`
- 会话集合、未读、置顶、免打扰等会话级状态优先落在 `stores/ConversationStore.ts`
- 消息拉取、发送、撤回、转发、pin、回执等消息级能力优先落在 `stores/MessageStore.ts` 或相关 store / utils
- 登录鉴权链路优先落在 `app/login.tsx`、`stores/AuthStore.ts`、`services/auth.ts`
- 配置型改动优先落在 `constants/NIMConfig.ts`、`app.json`
- 行为、流程、路由、运行时配置变化先判断是否需要 OpenSpec

如果一个功能同时影响页面和 store，先描述 store 状态怎么变，再描述页面如何消费这些状态。

## 输出要求

默认输出必须包含以下结构，除非用户明确要求别的格式。

## 功能说明
- 功能名称
- 目标
- 范围
- 非范围

## 核心实现路径
按真实代码路径说明：
1. 入口
2. 状态流转
3. 数据来源与写回
4. UI/交互更新
5. 异常与边界处理

每一项尽量带上明确文件路径或模块名，并解释为什么当前仓库要落在这里。

## 实施步骤
使用 3-7 个步骤描述最小落地顺序。

每一步至少包含：
- 修改点
- 涉及文件
- 完成标志

## 测试用例
至少覆盖以下类别：
- 正常流程
- 边界情况
- 异常流程
- 回归检查

每条测试用例尽量使用下面模板：

`[编号] [场景]`
- 前置条件：
- 操作步骤：
- 预期结果：

如果用户要求正式 spec，优先套用 `references/spec-template.md` 的章节结构。
如果用户要求测试设计或验收用例，优先套用 `references/testcase-template.md` 的用例模板。

## 当前仓库验证矩阵

默认按改动类型补充验证建议：
- 页面、路由、UI 组件：`npm run lint` + `npx tsc --noEmit` + 手动页面验证
- Store、NIM 配置、运行时配置：`npm run lint` + `npx tsc --noEmit` + `npx expo install --check` + 手动登录/会话/聊天验证
- OpenSpec / 工作流文档：`npm run format:check` + `OPENSPEC_TELEMETRY=0 openspec validate <change-name> --type change --no-interactive`

如果涉及登录、会话、聊天这三条主链路，手动验证中至少要点名这三条链路。

## 验收标准
- 功能完成的可观察结果
- 不通过的判定条件

## 风险与依赖
- 外部依赖
- 兼容性风险
- 可能阻塞点

## Spec 优先规则

如果用户要求“先落 spec”，先输出 spec 草案，不直接写实现代码。

Spec 至少包含：
- 背景
- 用户故事
- 功能范围 / 非范围
- 交互与状态规则
- 数据与接口影响
- 验收标准
- 测试用例

如果没有用户指定格式，就按 `references/spec-template.md` 输出。

如果测试用例、参考实现、现有代码三者冲突，优先级使用：
1. 用户明确指定的测试用例
2. 当前项目既有约束
3. 参考实现

## 实现阶段规则

当用户不是在讨论方案，而是在要求落地功能时：
- 不要只停留在分析
- 直接进入修改前的代码定位
- 先做最小闭环，再补边缘能力
- 每次改动前说明将修改的模块和原因
- 先对齐当前仓库已有实现模式，再决定是否引入新的抽象

## 测试设计规则

测试用例不要只写“验证成功/失败”。要覆盖：
- 用户主路径
- 空数据
- 非法输入
- 网络失败 / SDK 失败 / 权限失败
- 列表与详情联动
- 状态持久化或刷新后表现
- 与相邻功能的回归影响

如果仓库里已有 Excel、Spec、接口文档或 PRD 中的测试项，优先吸收并映射到当前功能。

如果仓库里存在成体系的测试清单，输出时要显式说明：
- 哪些测试点直接映射自现有测试用例
- 哪些测试点是为了补齐边界或回归风险新增的

## 示例

**示例 1：**
输入：参考某个聊天组件，把“消息转发”功能补到当前项目，先看当前仓库里相近能力怎么实现，再给我 spec 和测试点。

输出应至少包含：
- 当前仓库中消息相关能力落在哪些 route / store / utils
- 消息转发的功能边界
- 会话选择页、消息页、转发确认的核心实现路径
- 需要改动的 store / route / service
- 正常转发、取消转发、无可选会话、转发失败、转发后会话列表回归等测试用例

**示例 2：**
输入：基于当前仓库现有资料页和好友链路，给好友备注功能做一个最小实现方案。

输出应至少包含：
- 资料页入口
- 备注写入与本地刷新路径
- 会话列表昵称回填策略
- 修改成功、空备注、超长备注、刷新后持久化、列表页回归的测试用例

**示例 3：**
输入：把当前项目的登录实现方式提炼出来，告诉我后续短信登录类功能应该怎么按这个仓库的方式接入。

输出应至少包含：
- `app/login.tsx`、`stores/AuthStore.ts`、`services/auth.ts`、`stores/NIMStore.ts` 的职责分工
- 状态流转和错误处理路径
- 涉及配置文件和验证步骤
- 正常登录、验证码错误、账号不存在、初始化未完成、会话恢复等测试用例

## 执行提醒

在信息不足时，优先自己查代码、查文档、查现有实现；只有在关键验收口径无法推断时才提问。

始终避免这几种低质量输出：
- 只讲概念，不落到文件和状态流
- 没有测试用例
- 把多个功能混在一个方案里
- 忽略异常和回归场景

当仓库已经进入 Spec 驱动流程时，优先先补 Spec，再进入实现。
